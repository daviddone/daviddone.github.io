Android API Guides

Android 开发指南

简介 Introduction to Android

安卓提供了丰富的应用层框架来使得你能够在java开发环境下在移动设备下创建创新型App和游戏。左边导航栏中提供详情，利用Android丰富多样的API的创建App。如果你新入手Android，那么理解关于Android应用层框架的基本概念就显得尤为的重要。

应用提供多个入口

创建安卓应用由一系列可以独立调用的不同组件组合搞定。例如,一个Activity提供用户界面的的单个屏幕，一个Service独立的在后台运行。你可以采用Intent从一个组件到另外一个组件。甚至可以在不同应用之间调用组件。比如调用地图应用的Activity来显示地址。这种方式为单个应用提供多个入口。使得任何应用能够默认的调用其它的应用。

App适配不同的设备

Android提供可适配化的应用框架，为不同配置参数的设备提供唯一的资源。例如，你可以为不同大小的屏幕创建不同的layout布局文件。系统根据当前屏幕大小选择其中一种进行布局。你可以查询设备在运行时设备的特性是否可用，如果任何应用的特性要求使用指定硬件，比如说camera。若有必要，你可在google play等应用商城上，声明你的应用所需的特性不允许在不支持某种特性的设备上安装该应用。
了解更多：
Device Compatibility 设备兼容。
Resources Overview 资源概述。
User Interface Overview 用户界面概述。

应用的基础知识 Application Fundamentals

Android应用有JAVA语言编写，Android SDK工具将会编译所有代码（包括所有数据，资源文件）进入apk(安卓安装包)。apk安装包是一个压缩文件，以.apk后缀结尾。一个apk文件包含所有的Android应用内容，是Android设备用来安装App的文件。一旦安装到设备中，每一个Android应用在自己的安全沙盒中运行。

  • Android操作系统是一个多用户的Linux系统。每个App是不同的用户。
  • 每个进程都有自己的虚拟机（VM），因此，每个App的代码独立运行。
  • 默认情况下，每个App运行在它自己的Linux进程中，当App的任何组件需要被执行的时候，Android就会开启进程，当进程不在需要或者其它App需要系统恢复内存之时``，Android将会关闭进程。

通过这样的方式，Android系统实现了最少特权原则。也就是，每一个App在默认情况下，只能访问它本身运行所需的组件``,不再能够访问更多。这就能够确保，App不能访问哪些它没有权限访问的系统的其它部分，从而营造一种安全的环境。
然而，任然有一些方法可以让App和其它的App分享数据，让App获取系统的服务：

  • 两个App同时分享相同的Linux用户Id是可能的，当它们能够互相访问对方的文件。为了节省系统资源，拥有相同用户Id的App能够安排在一个Linux进程中运行,共享同一个虚拟机（应用程序也必须使用相同的证书）。
  • 一个应用程序可以申请获取设备数据,如用户的联系人，短信，挂载存储设备(SD卡)，摄像头,蓝牙,以及其它更多的设备数据。所有的App权限需要在用户安装是授予。
    以上包含了Android应用程序在系统中存在的相关基础知识。接下来的文档将会介绍:
  • 定义你的App的核心框架组件。
  • 为应用申明组件以及相关设备功能的清单文件（manifest）。
  • 与App代码分离开来的资源文件夹（Resources），它允许App为一系列的设备配置优化它的行为。

    应用组件

    应用程序组件是一个Android应用程序的基本构建模块，每个组件都是系统可以进入App的不同入口，并不是所有的组件都为用户实际的入口，有一写相互依赖，但是每一个都以自己的实体存在着并扮演一个特定角色 。每一个都是帮助定义应用程序的整体行为的独立构建模块。
    总共有四种不同类型的应用程序组件。每种类型有不同的目的，有着定义组件如何创建和销毁的生命周期。
    下列是四种应用程序组件:

    Activities（活动）

    一个Activity代表拥有单一屏幕的用户界面。例如，一个email应用程序可能拥有一个activity展示新邮件列表，另一个activity用来撰写Email，还有一个用来阅读email。尽管Activities相互协作，构成一个有凝聚力的用户体验，但是每一个都是独立存在。例如，不同的App能够启动这些Activities的任何一个（email应用允许的话）。例如，一个相机应用能够开启每个Email应用中撰写新邮件的Activity，这样来确保用户来分享一张图片。
    一个activity作为Activity的子类实现，你能在Activities的开发指南中了解更多。

    Services（服务）

    服务是一个在后台运行的组件,用来进行长时间运行的操作，抑或为远程进程工作。服务不提供用户界面，例如,一个服务可能在后台播放音乐，然后用户使用的是不同的应用程序，或许，它在不需要阻止用户正在进行的一个Activity的交互就能够通过网络获取数据。另一个组件,比如activity，可以通过可以开始服务,让它运行，抑或添加绑定来与之交互。
    service是作为Service的一个子类实现,你可以在service的官方文档中了解更多。

    Content providers（内容提供者）

    内容提供者管理一组共享App数据，你可以将数据存储在文件中，sqlite数据库，网络，或App能够访问的任何存储设备。通过内容提供者，其它应用程序可以查询或甚至修改数据（如果内容提供者允许它）。例如,Android系统提供了一个内容提供者来管理用户的联系人信息。因此,任何拥有适当的权限可以查询内容提供商的一部分（(如ContactsContract.Data)）来进行一个特定联系人的读写信息操作。
    内容提供商也用于读写 那些App私有，不能分享的相关数据。例如,记事本示例应用程序就使用一个内容提供者来保存笔记。

    Broadcast receivers（广播接受者）

    广播接收者是一个响应系统级范围的广播通知的组件。很多广播都来源于系统，例如屏幕关闭会发送广播，电池低会发送广播,或者照片被捕获也会发送广播。应用程序也可以启动广播，例如，可以用来——广播告知有些数据被下载到了设备上，通知它们可以被使用了。虽然广播接收器不显示用户界面，但是他们可以创建一个状态栏通知，用来提醒用户广播事件已发生了。更常见的,一个广播接收器是作为其他组件通道，被安排只做很小的工作`。例如，他可能初始化一个Service，Service在此基础上执行更多的操作。
    广播接受者通过集成BroadcastReceiver实现，每个广播作为一个Intent对象传递。更多相关信息,请参见BroadcastReceiver类。
    Android系统设计的一个独特之处在于任何应用程序都可以启动另一个应用程序的组件。例如,如果您希望用户通过摄像头获取一张照片。可能其他App有这个功能，你的App就可以使用，而不是，自己开发一个Acitvity来拍照。你不需要合并，甚至连相机应用程序的代码都不需要引用。而是，简单的开启相机应用中的一个Activity来获取一张照片。当完成之后，图片将会返回自己的App这样你就能够使用它了。对用户而言，好像相机功能实际上是本App的一部分一样。
    当系统启动一个组件，它会开启App的进程（如果它不是正在运行），初始化组件所需要的类。如果你的App启动相机应用的Acitvity来获取照片，在进程中所执行的Activity属于相机应用程序，而不属于你本身的App，这一点不想大多数其他系统上的应用程序。Android应用程序没有单一的入口点(没有Main函数)。
    因为系统把每个应用程序单独的运行在一个拥有文件权限的独立进程中，严格限制访问其他应用程序,应用程序不能直接激活另一个应用程序的组件。而Android系统是可以在这么做的。因此为了激活另一个应用程序的组件,您必须提供为系统发送一个消息,指定开启特定的组件的intent。然后系统为你激活的组件。

    激活组件

    四大组件中的3个，activities, services, and broadcast receivers都是通过叫做Intent的异步消息所激活的。Intent在运行时将各个组件相互绑定（你可以把Intents当作从其他组件请求的动作的使者`），无论是属于自己App的组件还是其他的应用的组件。
    Intent由一个Intent对象创建，它定义了激活其他指定控件或者一种特定类型的组件——Intent可以分别是显示的，或者是隐式的。
    对于activities和services，一个intent定义了用来执行的动作（如，到”视图”和发送）并有可能指定数据的URI来执行任务（被开启的组件可能需要被知道一些其他的信息）。例如,一个意图可能传达请求一个活动来显示一个图像或打开一个网页

如何掌握英文

目的

学习一门外语，不可避及的就是谈一谈，学习这门外语的目的，例如笔者学习英语的主要的目的是，学好英文，能够无障碍看美剧，看英文原著，看英文技术文档，用流利的英文交流，或者用英文来写写文章来陶冶一下情操。正是有了这些目标，我才能够有动力不断学习英文，有意识的运用英文。

方法

朗读和背诵

学习任何语言，最基本且有效的方法就是朗读和背诵。最开始的时候，一定要坚持，每天早上1小时，持续一年。背诵不必贪多，一篇一篇的文章背诵即可，对每篇文章的词组，语法都要达到熟练掌握。如此操作，只要能够达到50篇高级文章的熟练掌握，则会发现阅读会有很大的进步。

反复看和听

同种材料反复研看，研听，例如将 老友记 看两遍。充分利用自己的空余时间，把一些Adudio Book或是podcast下载到手机中，在没事干的时候戴上耳机反复听。这里推荐一些材料：《哈利波特》1-7，《乔布斯传》，《人性的弱点》，VOA news等的MP3音频都是不错的材料。

多用

俗话说，孰能生巧，如果学习一门外语，都不运用，怎么能够掌握呢。所以多开口讲英文，浏览英文网站，看英文原著，就会慢慢的扩大自己的词汇量。如果之前的反复阅读的方法算是精读的话，此方法就是略读了，例如笔者经常使用英文浏览技术网站。当遇到不认识的词语词组，采用翻译软件进行一下翻译即可，这里推荐一下国内的有道翻译软件。

提升中文能力

Android 屏幕适配

何为适配？


鉴于Android设备的种类繁多，屏幕大小各异，各种碎片化问题存在，在设计App的时候，需要将适配考虑进去。简单的说，适配就是为了让市面上使用各种屏幕大小，各种机型的用户们都能够正常，且流畅的使用我们的App。如果我们不考虑适配,直接使用UI设计师给出长度为50px方形的logo图形，在Android开发的布局文件中设置控件长宽为50px，那么我们得到的结果是任何设备上显示的控件长度50px，就会出现在低分辨率的手机上logo会失真，高分辨率会缩小。

基本知识介绍

px

px是pixel的缩写，即像素，它是数字图像在计算机图形学中基本元素，即屏幕上的最小单位。通常用来指代电子设备的分辨率：
如我们所说的分辨率1280 × 720，实际上就是竖屏上面总共的像素点1280,横屏上面像素点为720个。

inch

inch，英寸，屏幕长度计量单位，一英寸等于2.54厘米。一般常说的5英寸大小的手机是指手机的对角线是5英寸，12.7cm长。

dpi

dot per inch，每英寸上面的像素点，即Android手机的屏幕密度。对于4.5英寸，1280 × 720分辨率的手机，我们可以通过对角线上的分辨率除以屏幕大小4.5英寸得到dpi。注意：对角线分辨率通过长1280,宽720做勾股定理算出。

算出dpi为326,非常接近320。参考官方的屏幕密度表：

dip

dip和dp一样，device independent pixels，即是设备独立像素，是Android设备的特有的屏幕计量单位，按照标准在160dpi的设备下，1px = 1dp。其余情况，需要换算，采用公式计算而得： px = dp × (dpi / 160) (标准情况是160)。这里的dpi/160即是屏幕密度的比例值。如在320的屏幕密度下1px换算成dp为：dp = 1/(320/160) = 0.5。

sp

ScaledPixels，即是放大像素。在Android官方网站上并没有px转换为sp的公式，但是sp的大小和dp相似，所以可以采用px = sp × (dpi/160)，则sp = px/(dpi/160)来做计算。
官方教材

选择xhdp作为主适配

目前市面上的高端机的分辨率范围大多分辨率为1280 × 720，所以优先考虑分辨率为1280 × 720作为主要适配的机型。这时候可以协调公司的UI设计师在分辨率为1280 × 720的画布上作图，此时屏幕密度值为320，则将切好的图标放在xhdpi资源文件夹目录下。同理如果设计稿画布分辨率对应的屏幕密度在240左右，则将图片放在hdpi路径下（具体关系参照上图）。这个时候直接在布局文件中采用wrap_content引用即可，当然你也可以将图片的大小换算成dp值，采用图标px大小除以屏幕密度比例值即为dp值（dp =px/(dpi /160)）。在Android开发中我们采用dp来代替px作为尺寸大小单位，是因为px在任何屏幕上的大小是一致的，而dp则是根据每个手机屏幕密度值变化的，在低屏幕密度值的情况下，控件显示的长度小一些，而高屏幕密度值，控件显示长度则大一些。
